hmdccondor.HMDCCondor module¶

HMDCCondor module provides most-commonly-used functions for interacting with HTCondor clusters: polling for job status, acquiring schedd objects, and submitting jobs according to the HMDC ClassAd schema. This replaces Condor.pm in the hmdc-admin repository.

class hmdccondor.HMDCCondor.HMDCCondor[source]¶

HMDCCondor provides an interface to a local HTCondor cluster. It instantiates a local HTCondor schedd object for all operations. The initialization method for the HMDCCondor class acquires a local HTCondor schedd object, sets the polling timeout, and local LDAP server.

Note

self.ldap_server and self.ldap_base_dn should be acquired via rceapp.yml rather than statically defined in __init__()

poll(jobid, return_status=None, use_local_schedd=False, want_results=[2])[source]¶

poll() takes a job id and returns a (JobStatus,ClassAd) tuple if, before the polling timeout window exceeds, a job has a JobStatus classad element which matches an integer in the return_status array. It then compares this JobStatus to the want_results array. If the job polled has a JobStatus which is also in the want_results array, poll succeeds, otherwise, raises an exception.

Parameters:
  • jobid (int) – job id
  • return_status (list) – an array of JobStatus’s from which polling should return.
  • use_local_schedd (bool) – If true, uses local schedd, otherwise finds schedd for jobid.
  • want_results (list) – An array of JobStatus values that are compared to the JobStatus of the successfully returned job.
Returns:

(JobStatus, ClassAd) of polled job
Return type:

tuple
Raises:

hmdccondor.HMDCExceptions.RCEJobDidNotStart
Raises:

hmdccondor.HMDCExceptions.RCEJobTookTooLongStartError
Example: 
>>> from hmdccondor import HMDCCondor, CONSTANTS
>>> HMDCCondor().poll(73, return_status=[2], use_local_schedd=True,
  want_results = [ CONSTANTS.JOB_STATUS_RUNNING ])

hmdccondor.HMDCCondor.poll_thread(id, return_status, use_local_schedd)[source]¶

poll_thread is a function which when provided as an argument to apply_async() is executed in an independent thread. poll_thread creates a new HMDCPoller which runs until the job represented by id has a JobStatus equal to any number in the return_status array.

Parameters:
  • id (int) – HTCondor Job Id to poll for
  • return_status (list) – An array of JobStatus values to look for.
  • use_local_schedd (boolean) – If set to true, uses the schedd of the executing host. If false, finds the schedd whcih submitted the HTCondor job represented by id.
Returns:

(JobStatus, ClassAd) or (None,None) if no such job exists.
Return type:

tuple
Example: 
>>> import hmdccondor
>>> from hmdccondor.HMDCCondor import poll_thread
>>> poll_thread(10, [1,2,3,4,5], True)

hmdccondor.HMDCWrapper module¶

HMDCWrapper class is responsible for setting the resource limits of a job and executing the job’s command as specified in the job’s ClassAd in the Cmd field. HMDCWrapper reads the _CONDOR_JOB_AD and _CONDOR_MACHINE_AD environment variables which are present in any application executing within a HTCondor job slot to read in the appropriate classads and execute the application. Note that wrappers in HTCondor must always end with an exec system call.

Note

According to HTCondor documentation:

The following variables are set in the environment of the the
USER_JOB_WRAPPER script by the condor_starter daemon, when the
USER_JOB_WRAPPER is defined.

_CONDOR_MACHINE_AD
The full path and file name of the file containing the machine
ClassAd.
_CONDOR_JOB_AD
The full path and file name of the file containing the job ClassAd.
_CONDOR_WRAPPER_ERROR_FILE
The full path and file name of the file that the USER_JOB_WRAPPER
script should create, if there is an error. The text in this file
will be included in any HTCondor failure messages.
